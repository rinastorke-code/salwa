# نظام إدارة الموارد البشرية — مديرية الشؤون الاجتماعية والعمل

تطبيق PWA داخلي خفيف، عربي/إنجليزي (RTL افتراضياً)، Next.js 14 + Supabase.

## التشغيل
```bash
npm install
cp .env.example .env.local   # املأ مفاتيح Supabase الثلاثة
npm run dev
```

## قاعدة البيانات (نفّذ بالترتيب في SQL Editor)
1. `01_schema.sql`      — الجداول والفهارس
2. `02_triggers.sql`    — (يُلغى محتواه لاحقاً بـ 06؛ يُترك للتوافق)
3. `03_functions.sql`   — دوال RPC (نقل، بحث، إحصاءات)
4. `04_rls.sql`         — سياسات الأمان + مخزن الوثائق
5. `05_seed.sql`        — بيانات تجريبية (اختياري)
6. `06_ledger.sql`      — نموذج الرصيد المحسوب + أيام العمل + record_leave
7. `07_activity_log.sql`— سجل النشاط + محفّزات التدوين
8. `08_rbac_crud.sql`   — دوال التعديل/الحذف حسب الدور

ثم أنشئ مستخدمي Auth واربطهم في `users_roles` (مدير عام + مساعدو إدخال).

## نموذج الرصيد (Ledger) — الأهم
الإجازات سجلات غير قابلة للتغيير، والرصيد **محسوب عند القراءة** من
`v_employee_balance`. لذلك حذف/تعديل أي إجازة يصحّح الرصيد تلقائياً.
- `working_days()` يحسب أيام العمل ويستثني العطلة الأسبوعية (قابلة للضبط في
  `app_settings.weekend_dows`) والعطل الرسمية في جدول `holidays`.
- `record_leave()` يمنع الرصيد السالب، مع تجاوز للمدير العام فقط.

## الاستيراد الذكي من Excel
`/api/import-employees` يقرأ العناوين العربية، يحوّل الأرقام الهندية، يطبّع
الأسماء والتواريخ، يحلّ أسماء المواقع (يطابق الموجود وينشئ الناقص)، ويعيد
تقريراً بالصفوف المرفوضة وأسبابها. منطق التحليل في `src/lib/import.ts` (مُختبَر).

## سجل النشاط
كل إضافة/تعديل/حذف/نقل/استيراد يُسجَّل تلقائياً في `activity_log` عبر محفّزات
قاعدة البيانات (لا يعتمد على الواجهة)، ويظهر في صفحة «النشاط».

## ملاحظة الخطوط العربية في PDF
سجّل خط TTF (Amiri/Cairo) عبر `doc.addFont` في `src/lib/pdf.ts` لإظهار العربية.

## ما تم التحقق منه فعلياً (Postgres حقيقي عبر PGlite)
- تسلسل الملفات 01→08 يُحمّل نظيفاً على تثبيت جديد.
- أيام العمل، الرصيد المحسوب، تصحيح الرصيد عند الحذف، منع السالب، التجاوز الإداري.
- سجل النشاط يُكتب بملخصات عربية صحيحة.
- محلّل Excel: أرقام هندية، مسافات، مكرر، حقول ناقصة، طول غير صالح، تواريخ.

## التحديث الكبير: المراكز والتفقّد اليومي (الملفات 09–10)

### نموذج الإجازات النهائي
- نوعان عند الإدخال: **إدارية** (تُخصم) أو **مرضية** (لا تُخصم، بلا سقف).
- الإدارية: أيام (قائمة منسدلة) أو ساعات (قائمة منسدلة، 8 ساعات = يوم).
- **السقف السنوي حسب الموقع** (`locations.annual_cap`) مع تجاوز فردي
  (`employees.annual_cap_override`). الافتراضي 30. يضبطه الأدمن من الإعدادات.
- تجاوز السقف لا يُرفض، بل يُسجَّل **بانتظار الاعتماد** مع علامة `over_cap`.

### بوابات المراكز (معزولة)
- الأدمن يولّد لكل مركز حساباً (اسم + كلمة سر bcrypt) من الإعدادات.
- جلسة المركز عبر **كوكي موقّع (JWT/jose)** — لا Supabase Auth، بسيطة وخفيفة.
- **العزل:** كل واجهات المركز تمرّ عبر API على الخادم يَفرض `location_id` من
  الجلسة (لا من المتصفح إطلاقاً)، ويرفض أي معرّف عامل لا يخص المركز. كما أن
  RLS يمنع الوصول المباشر. المركز لا يضيف موظفين (الإضافة للأدمن حصراً).
- المركز: تفقّد يومي (حاضر/غائب + سبب) بأسماء محفوظة، ورفع إجازات.

### التفقّد → إجازة تلقائياً
أي «غائب — إجازة إدارية» يُنشئ إجازة إدارية (يوم كامل، أو ساعات إن حُدّدت)
بحالة pending. «مرضية» تُسجَّل دون خصم. «إذن» تُسجَّل في التفقّد فقط.

### الاعتماد والإشعارات
- رفوعات المراكز تُحسب فوراً (pending) ويعتمدها **أي موظف مركزي** من صفحة
  «الاعتمادات» (confirm/reject).
- **تنبيه المراكز المتأخرة** عن التفقّد بعد 10 صباحاً (عدا الجمعة/السبت) عبر
  `late_centers()`، مع **إشعار PWA محلي** وبانر في اللوحة.
- سجل النشاط يشمل **دخول المراكز** والتفقّد والاعتمادات.
